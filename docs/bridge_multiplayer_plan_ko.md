# 멀티플레이 컨트랙트 브릿지 구현 목표

이 문서는 GIU 사이트에 컨트랙트 브릿지 멀티플레이 기능을 추가할 때의 제품 방향과 구현 순서를 기록합니다.

## 목표

- 기존 게임 약속 기능을 브릿지 로비의 진입점으로 활용합니다.
- 약속 참여자가 4명일 때 브릿지 테이블을 열고, North/East/South/West 좌석을 배정할 수 있게 합니다.
- 첫 구현은 턴 기반 멀티플레이에 집중하고, 실시간성은 현재 Next.js, Prisma, PostgreSQL, Railway 구조와 충돌이 적은 방식으로 시작합니다.
- 카드 디자인은 `docs/playing-card-design-preview.html`의 미니멀 카드 방향을 기반으로 React 컴포넌트화합니다.

## 현재 환경 판단

- 현재 프로젝트는 Next.js App Router, React, Prisma, PostgreSQL, Railway standalone 배포 구조입니다.
- WebSocket, Redis, 별도 pub/sub 서비스는 아직 없습니다.
- 컨트랙트 브릿지는 턴 기반 게임이므로 초기 버전은 DB 이벤트 로그와 polling 또는 SSE로도 구현 가능합니다.
- 멀티 인스턴스 확장이나 대규모 동시 접속이 필요해지면 WebSocket 또는 별도 실시간 서비스 도입을 다시 검토합니다.

## 현재 구현 상태

- 현재 단계: 정식 비딩, 플레이, 복수 딜, 점수, 결과 기록, 관전, 수동 만료, 통합 테스트까지 주요 계획을 구현했습니다. 다음 단계는 실제 운영 지표 관찰과 자동 정리 필요성 판단입니다.
- 구현 완료: 브릿지 약속 생성, 방 생성, 약속 카드 전체 클릭 입장, 카드 클릭 자동 참여, 랜덤 좌석 배정, 방장 좌석 섞기, 방장 좌석 직접 변경/교환, 로비 참여자 나가기/내보내기, 내보내기 대상 자동 홈 이동, 딜 시작 시 자동 좌석 확정, 개인 손패 표시, 정식 비딩 코어, 패스/입찰/더블/리더블 검증, 자동 컨트랙트 확정, 카드 플레이, follow suit 검증, 트릭 승자 계산, 더미 공개 타이밍, 획득/필요 트릭 표시, 라운드 종료 결과 저장, 취약/더블 기준 점수 팝업, 복수 딜 저장 구조, 딜러/취약 순환, 보드별 결과 목록, 세션 NS/EW 누적 점수.
- 실시간 동기화: `BridgeEvent` 이벤트 로그, SSE route, 2.5초 polling fallback을 적용했습니다.
- 운영 안전장치: 방장/관리자는 브릿지 방 또는 약속 관리 화면에서 방을 완료/취소할 수 있고, 삭제된 방 접근은 404 대신 메인 화면으로 이동합니다.
- Docker 테스트: `docker-compose.test.yml` 기준으로 PostgreSQL과 Next 앱이 구동되며, 현재 브릿지 migration은 `20260612000300_bridge_spectators`까지 적용됩니다.
- 다음 핵심 방향: SSE 연결 수, polling 부하, 방별 이벤트 증가량, 실제 방 지속시간을 관찰하고 자동 만료 또는 별도 realtime 서비스가 필요한 시점을 판단합니다.

## 권장 구현 순서

1. 약속 기반 브릿지 로비
   - 기존 `Meetup`을 브릿지 게임 생성의 진입점으로 사용합니다.
   - 약속 생성 시 게임 종류를 일반 보드게임 약속과 브릿지 약속으로 구분할 수 있게 확장합니다.
   - 브릿지 약속은 현장에서 바로 테이블을 열 수 있도록 시작 시간의 미래 시각 검증을 적용하지 않습니다.
   - 브릿지 약속은 생성 즉시 브릿지 방으로 이동하고, 참여자는 빈 좌석에 자동 배정됩니다.
   - 일반 약속도 생성 직후 약속 관리 화면으로 이동합니다.
   - 현재 구현: `Meetup.kind`, `BridgeRoom`, `BridgeSeat` 기반으로 브릿지 약속 생성 즉시 방 생성, 참여/취소 시 좌석 동기화, 좌석 표시 페이지까지 연결했습니다.
   - 현재 구현: 약속 목록에서는 별도 입장/참여 버튼 대신 약속 카드 본문 전체를 눌러 자동 참여 후 브릿지 방 또는 약속 상세로 들어갑니다.

2. 브릿지 방과 좌석
   - `BridgeRoom`과 `BridgeSeat` 모델을 추가합니다.
   - 한 방에는 North, East, South, West 네 좌석만 허용합니다.
   - 같은 사용자가 한 방에서 여러 좌석을 차지하지 못하게 DB 제약과 서버 검증을 둡니다.
   - 현재 구현: 참여자는 빈 좌석 중 랜덤으로 배정되고, 방장/관리자는 로비에서 좌석을 섞을 수 있습니다.
   - 현재 구현: 방장/관리자는 딜 시작 전 로비에서 특정 참여자를 좌석으로 직접 이동하거나, 이미 사람이 앉은 좌석과 교환할 수 있습니다.
   - 현재 구현: 딜 시작 전 로비에서 참여자는 방을 나갈 수 있고, 방장/관리자는 참여자를 내보낼 수 있습니다. 내보내기 대상은 방 화면에서 자동으로 홈 화면으로 이동합니다.
   - 현재 구현: 딜 생성 시 좌석을 한 번 더 자동 섞어 시작 방향을 확정합니다.

3. 딜 생성과 손패 표시
   - 서버에서 52장 덱을 셔플하고 4명에게 13장씩 배분합니다.
   - 각 사용자는 자기 손패만 받을 수 있어야 합니다.
   - 관리자/디버그 화면이 아닌 곳에서는 다른 플레이어의 손패를 내려주지 않습니다.
   - 현재 구현: `BridgeDeal.hands` JSON에 좌석별 손패를 저장하고, `/bridge/[id]`에서 현재 사용자의 좌석 손패만 카드 UI로 표시합니다.

4. 컨트랙트 확정 후 플레이
   - 초기 버전에서는 비딩을 생략하고 방장이 컨트랙트와 선언자를 선택했습니다.
   - 현재 버전에서는 딜 생성 후 정식 비딩 코어를 거쳐 컨트랙트와 선언자를 자동 확정합니다.
   - 서버에서 follow suit, 턴 순서, 트릭 승자를 검증합니다.
   - 더미 공개와 트릭 진행을 우선 구현합니다.
   - 현재 구현: 비딩 종료 후 더미와 첫 리더가 정해지고, 각 플레이는 서버 액션에서 현재 턴, 카드 소유권, follow suit를 검증합니다.
   - 현재 구현: `BridgeTrick`, `BridgePlay`에 카드 플레이를 기록하고, 4장이 모이면 서버에서 트릭 승자를 계산해 다음 턴을 승자에게 넘깁니다. 더미 차례에는 선언자가 더미 카드를 낼 수 있습니다.
   - 현재 구현: 더미 손패는 컨트랙트 확정 직후 숨겨두고, 선플레이어가 첫 카드를 낸 뒤 공개합니다. 진행 상태에는 선언자 팀의 획득 트릭/필요 트릭을 표시하고, 13트릭이 끝나면 라운드 결과 팝업을 띄웁니다.
   - 현재 구현: 13트릭 완료 후 `BridgeDeal`에 선언자 트릭, 수비 트릭, 성공/실패, 초과/부족 트릭, 점수를 저장합니다. 방은 다음 딜을 시작하거나 세션을 종료할 수 있도록 `PLAYING` 상태를 유지합니다.
   - 현재 구현: 새 딜은 비딩부터 시작하며 패스/입찰/더블/리더블 후 3패스로 자동 컨트랙트를 확정합니다. 네 명이 모두 패스하면 패스아웃으로 0점 종료합니다.
   - 현재 구현: 정식 비딩을 우회하던 `setBridgeContractAction`은 삭제했습니다. 운영 복구 기능이 필요하면 별도 관리자 기능으로 다시 설계합니다.

5. 비딩과 점수 계산
   - 딜러, 취약 상태, 콜 순서, 패스/더블/리더블 검증을 추가합니다.
   - 컨트랙트 결정 후 플레이 단계로 전환합니다.
   - 컨트랙트 결과와 점수를 저장합니다.
   - 현재 구현: `BridgeCall`에 콜 기록을 저장하고, 현재 비딩 턴, 상위 입찰, 더블/리더블 가능 조건, 3패스 종료, 패스아웃을 서버에서 검증합니다.
   - 현재 구현: 선언자는 최종 계약 팀에서 해당 무늬를 처음 부른 좌석으로 자동 결정합니다.
   - 현재 구현: 취약 여부와 더블/리더블을 반영해 계약 성공/실패 점수를 계산합니다.
   - 현재 구현: 보드 번호 기준 표준 16보드 취약 순환을 적용하고, 선언 측의 취약 여부를 점수 계산에 사용합니다.

6. 방 상태 실시간 동기화
   - 해결한 문제: 새 참여자가 들어오거나 딜이 생성되어도 이미 방을 보고 있는 다른 사용자 화면이 자동 갱신되지 않던 문제를 우선 처리했습니다.
   - 원인: 이전 구현은 Server Action과 `revalidatePath` 중심이라, 요청을 보낸 사용자 화면은 갱신되지만 다른 브라우저 세션으로 상태 변경 이벤트를 밀어주지 않았습니다.
   - 1차 해결책은 짧은 주기의 polling입니다. `/bridge/[id]`에서 2-3초마다 방 상태 API를 조회하거나 `router.refresh()`를 호출해 좌석, 방 상태, 딜 생성 여부를 갱신합니다.
   - 현재 구현: `BridgeEvent` 이벤트 로그와 `/bridge/[id]/events` SSE 스트림을 추가했습니다. 서버 액션은 방 생성, 좌석 참여, 좌석 이탈, 딜 생성을 이벤트로 기록하고, `BridgeRoomSync`는 이벤트 수신 시 `router.refresh()`를 호출합니다.
   - fallback 구현: SSE를 사용할 수 없거나 연결이 끊겨도 `BridgeRoomSync`가 2.5초마다 `router.refresh()`를 호출하고, 브라우저 탭이 다시 보일 때도 갱신합니다.
   - WebSocket은 방 수와 동시 접속이 많아지거나 양방향 실시간성이 더 필요해질 때 검토합니다. 현재 Railway/Next.js/PostgreSQL 구조에서는 polling 또는 SSE가 더 작고 안전한 시작점입니다.

7. 재접속, 관전, 운영 기능
   - 사용자가 새로고침하거나 재접속해도 현재 방 상태를 복구합니다.
   - 관전자는 모든 공개 정보만 볼 수 있게 합니다.
   - 관리자 로그와 정리 기능을 추가합니다.
- 현재 안전장치: 방장/관리자는 브릿지 방 화면과 약속 관리 화면에서 방을 완료 또는 취소할 수 있고, 약속 삭제 cascade로 연결된 `BridgeRoom`, `BridgeSeat`, `BridgeDeal`, `BridgeEvent`가 함께 정리됩니다.
- 현재 안전장치: 방 취소는 오류 화면 대신 팝업 메시지를 보여주고 초기 화면으로 이동하는 폼 흐름을 사용합니다.
- 현재 안전장치: 로비 30분, 플레이룸 3시간을 기준으로 관리자 수동 `EXPIRED` 처리와 읽기 전용 기록 보존을 적용했습니다.
- 후속 판단: 실제 운영 데이터를 확인한 뒤 cron 또는 요청 시 lazy cleanup 방식의 자동 만료가 필요한지 결정합니다.

## 서버 검증 원칙

- 카드 셔플, 딜, 비딩, 카드 플레이, 점수 계산은 클라이언트 값을 신뢰하지 않고 서버에서 검증합니다.
- 게임 상태 변경은 트랜잭션으로 처리하고, 성공한 행동은 이벤트 로그로 남깁니다.
- 클라이언트 동기화는 `BridgeEvent` 같은 append-only 이벤트를 기준으로 합니다.
- 각 API 응답은 현재 사용자에게 허용된 정보만 포함합니다.

## 우선 개선점

- 완료: 브릿지 방 자동 갱신은 `BridgeEvent`, SSE, polling fallback으로 처리했습니다. 참여자 입장, 좌석 변경, 딜 생성, 컨트랙트 확정, 카드 플레이, 트릭 완료가 모든 접속자 화면에 반영됩니다.
- 완료: 규칙에 맞지 않는 카드 플레이는 서버 액션에서 차단하고, 사용자에게는 팝업 경고로 보여줍니다.
- 완료: 브릿지 테이블 중앙에 현재 트릭에서 각 좌석이 낸 카드를 표시하고, 트릭 완료 시 승자 팀을 표시합니다.
- 완료: 더미 손패는 선플레이어가 첫 카드를 내기 전까지 공개하지 않습니다.
- 완료: 13트릭 완료 후 결과를 DB에 저장하고, 방/딜 상태를 완료 상태로 전환합니다.
- 완료: 컨트랙트 성공/실패, 초과/부족 트릭, 취약 여부, 더블/리더블 기준 점수 계산을 추가했습니다.
- 완료: 정식 비딩 코어를 추가했습니다.
- 완료: 약속 카드 전체 클릭 자동 참여, 랜덤 좌석 배정, 방장 좌석 섞기, 방장 좌석 직접 변경/교환, 로비 참여자 나가기/내보내기, 내보내기 대상 자동 홈 이동, 내 좌석 하단 기준 테이블 회전, 선언자 별 표시, 취약 상태 붉은 강조를 추가했습니다.
- 완료: 복수 딜, 딜러 순환, 보드 번호 기준 취약 순환, 세션 누적 점수를 추가했습니다.
- 완료: 핵심 서버 도메인 통합 테스트, 제한시간 만료 방 정리, 완료·만료 세션 기록 목록, 관전 정책을 구현했습니다.
- 운영 고려: 여러 Railway 인스턴스로 확장하기 전까지는 DB 이벤트 로그 기반 polling/SSE로 충분하며, 확장 시 Redis pub/sub 또는 외부 realtime 서비스를 검토합니다.
- 방 정리 고려: 방장/관리자에게 즉시 완료/취소 버튼을 제공하고, 오래 열린 로비 또는 플레이룸은 제한시간 만료 후 목록에서 제거하거나 비활성 처리합니다.

## 구현 완료 및 운영 관찰 항목

### 1순위: 핵심 게임 흐름 통합 테스트

- 서버 액션 기준으로 정상 비딩 종료, 패스아웃, 더블/리더블, 13트릭 완료, 다음 딜 생성, 세션 종료를 검증합니다.
- 잘못된 턴, revoke(follow suit 위반), 다른 좌석 카드 제출, 완료된 딜 중복 요청, 동시에 들어온 요청을 차단하는지 검증합니다.
- 현재 단위 테스트만 있는 점수/취약 로직 외에 DB 상태 전이와 이벤트 생성까지 확인합니다.
- 현재 구현: PostgreSQL 테스트 DB에서 정상 비딩 종료, 패스아웃, 잘못된 차례, 더블/리더블, 잘못된 더블 rollback, 동시 중복 콜을 검증합니다.
- 현재 구현: 카드 소유권과 턴, 더미 제어, follow suit 위반 rollback, 트릭 승자, 13트릭 완료, 점수·이벤트·운영 로그 저장을 검증합니다.
- 현재 구현: 다음 딜의 보드 번호·딜러·취약 순환·좌석 유지와 진행 중 딜의 중복 생성/세션 종료 차단, 정상 세션 종료를 검증합니다.
- 완료: 참여자 내보내기 성공, 일반 참여자 권한 차단, 딜 시작 후 이탈 차단, 방장 이탈 차단을 PostgreSQL 통합 테스트에서 검증합니다.

### 2순위: 운영용 강제 컨트랙트 액션 정리

- 완료: UI에서 사용하지 않고 정식 비딩을 우회하던 `setBridgeContractAction`을 제거했습니다.
- 운영 복구 기능이 필요해지면 관리자 전용 화면, 실행 사유, 운영 로그를 갖춘 별도 기능으로 다시 설계합니다.

### 3순위: 완료 세션 결과 접근성

- 완료: 참여자가 내 페이지의 `내 브릿지 기록` 목록에서 완료·만료 세션과 최종 NS/EW 점수를 확인하고 기록 화면으로 다시 들어갈 수 있습니다.
- 완료: 결과 화면의 보드별 목록에 계약, 선언자, 취약 상태, 성공/다운, NS/EW 점수를 표시합니다.
- 완료: 운영자는 약속 관리 화면에서 최근 완료 브릿지 세션을 종료 시각과 함께 조회할 수 있습니다.
- 현재 제약: 방 취소는 약속과 방을 cascade 삭제하므로 취소 세션 자체는 목록에 남지 않고 운영 로그에서만 확인할 수 있습니다.

### 4순위: 만료와 정리 정책

- 완료: 로비 30분, 플레이 중 3시간을 기본 제한으로 두고 마지막 `BridgeEvent` 또는 방 갱신 시각을 기준으로 만료 대상을 계산합니다.
- 완료: `EXPIRED` 상태와 관리자 수동 만료 버튼, 서버 기준시간 검증, 운영 로그를 추가했습니다.
- 완료: 만료된 방은 홈 진행 목록에서 숨기고, 참가자와 관리자는 기록 목록 및 직접 링크에서 읽기 전용으로 확인할 수 있습니다.
- 다음: 실제 운영 데이터를 관찰한 뒤 제한시간을 조정하고, lazy cleanup 또는 Railway cron 자동 만료를 검토합니다.

### 5순위: 관전 기능

- 완료: 관전은 기본 차단이며 방장 또는 관리자가 진행 중 방에서 허용하거나 다시 차단할 수 있습니다.
- 완료: 비참가자는 관전 허용 방에서 비딩, 공개된 더미, 현재 트릭, 완료 결과만 보고 비공개 손패와 액션 UI는 받지 않습니다.
- 완료: 관전자 SSE에는 이벤트 payload를 제외하고 화면 갱신 신호만 전송합니다.
- 완료: 딜이 시작된 관전 허용 방은 홈 약속 카드에 `관전 가능`으로 표시하고 자동 참여 대신 관전 화면으로 이동합니다.
- 완료: 관전 설정 변경은 `BridgeEvent`와 운영 로그에 기록합니다.

### 6순위: 실시간 운영 강화

- 현재 SSE와 2.5초 polling fallback의 연결 수, DB 이벤트 증가량, Railway 인스턴스 환경에서의 안정성을 관찰합니다.
- 완료: 진행 중인 방은 최근 `BridgeEvent` 500개만 유지하고, 완료·만료 방은 이벤트를 모두 삭제합니다.
- 다중 인스턴스 또는 동시 방 증가 시 Redis pub/sub 또는 외부 realtime 서비스 도입을 검토합니다.

## 구현 단계 이력

### 1차: 복수 딜 기반으로 전환

- `BridgeRoom`과 `BridgeDeal`의 1:1 구조를 1:N 구조로 확장합니다.
- `BridgeRoom.deal` 단수 relation을 `BridgeRoom.deals`로 바꾸고, 현재 진행 중인 딜은 `BridgeDeal.completedAt IS NULL` 또는 별도 `currentDealId` 방식 중 하나로 식별합니다.
- `BridgeDeal`에 `boardNumber`를 추가하고 방 안에서 1번 보드부터 순서대로 생성합니다.
- 기존 단일 딜 데이터는 migration에서 `boardNumber = 1`로 보존합니다.
- 완료 기준: 기존 단일 딜 방이 깨지지 않고, 새 방에서 1번 보드 생성/비딩/플레이/결과 저장이 기존과 동일하게 동작합니다.
- 현재 구현: `BridgeRoom.deals`, `BridgeDeal.boardNumber`, `@@unique([roomId, boardNumber])`로 전환했고 현재 딜은 완료되지 않은 딜을 우선 사용합니다.

### 2차: 딜러와 취약 순환

- 보드 번호 기준 딜러 순환을 적용합니다. 1번 North, 2번 East, 3번 South, 4번 West 순서로 반복합니다.
- 보드 번호 기준 취약 상태를 적용합니다. 정식 duplicate bridge 취약표를 함수로 구현하고, 딜 생성 시 `BridgeDeal.vulnerability`에 저장합니다.
- 방 화면의 진행 상태에는 보드 번호, 딜러, 취약 팀을 함께 표시합니다.
- 완료 기준: 1-16번 보드의 딜러/취약 값이 표와 일치하고, 점수 계산이 저장된 취약 상태를 사용합니다.
- 현재 구현: 보드 번호 기준 딜러/취약 함수를 추가했고, 딜 생성 시 저장된 취약 상태를 점수 계산과 화면 표시에 사용합니다.

### 3차: 세션 누적 점수와 다음 딜 진행

- 라운드 완료 후 `BridgeRoom.status`를 바로 `COMPLETED`로 바꾸지 않고, 다음 딜 대기 상태를 유지합니다.
- 방장/관리자는 라운드 결과 확인 후 `다음 딜 시작` 또는 `세션 종료`를 선택할 수 있습니다.
- 방 화면에는 이전 딜 결과 목록과 North/South, East/West 누적 점수를 표시합니다.
- 패스아웃도 0점 결과로 딜 목록과 누적 점수에 포함합니다.
- 완료 기준: 한 방에서 최소 2개 이상의 딜을 연속 진행할 수 있고, 새로고침/재접속 후에도 현재 딜과 누적 점수가 복구됩니다.
- 현재 구현: 라운드 완료 후 방 상태를 `PLAYING`으로 유지하고, 방장/관리자가 `다음 딜 시작`으로 새 보드를 생성할 수 있습니다. 화면에는 완료 보드 결과와 NS/EW 누적 점수를 표시합니다.

### 4차: 결과 화면과 방 정리

- 완료된 방은 기본 약속 목록에서 숨기거나, 별도 결과 화면으로 이동시키는 정책을 확정합니다.
- 방장/관리자 완료/취소 버튼의 의미를 `세션 종료`와 `방 취소`로 더 명확히 정리합니다.
- 오래 열린 로비와 플레이룸의 제한시간 정책을 적용합니다. 첫 단계는 화면 안내와 관리자 수동 정리, 이후 lazy cleanup 또는 cron으로 확장합니다.
- 완료 기준: 진행 중인 방, 종료된 방, 취소된 방이 홈/관리 화면에서 서로 혼동되지 않습니다.
- 현재 구현: 브릿지 세션 종료는 약속 삭제 대신 `BridgeRoom.status = COMPLETED`로 남기고, 방 화면은 보드 결과와 최종 NS/EW 누적 점수를 중심으로 표시합니다.
- 현재 구현: 진행 목록에는 `LOBBY`, `PLAYING` 브릿지 방만 노출하고, 완료·만료 세션은 내 페이지와 관리자 약속 관리의 기록 목록에서 다시 접근할 수 있습니다.

### 5차: 테스트와 운영 검증

- 딜러/취약 순환 함수와 점수 계산 함수에 단위 테스트를 추가합니다.
- 현재 구현: 딜러/취약 순환과 취약/비취약 계약 성공·실패 점수를 `src/lib/bridge-scoring.test.ts`에서 검증합니다.
- 규칙 확인: duplicate bridge에서는 선언 측 기준 plus/minus 점수를 사용합니다. 계약 실패 시 선언 측 음수와 수비 측 동일 절댓값의 양수로 기록하며, 취약 상태는 선언 측의 게임 보너스와 다운 페널티 등에 반영합니다.
- 현재 구현: 서버 도메인 함수 기준으로 비딩 종료, 패스아웃, 동시 콜, 카드 플레이, 13트릭 완료 시나리오를 PostgreSQL 통합 테스트에서 검증합니다.
- 현재 구현: 다음 딜 생성과 세션 종료 상태 전이까지 PostgreSQL 통합 테스트에서 검증합니다.
- 완료: 참여자 내보내기와 로비 이탈 권한 시나리오를 PostgreSQL 통합 테스트에서 검증합니다.
- Docker 테스트 환경에서 migration, seed, dev server 부팅을 한 번에 확인합니다.
- 완료 기준: `npx tsc --noEmit`, `npm run lint`, Prisma migration 적용, 핵심 브릿지 시나리오 수동 검증이 통과합니다.

## 예상 DB 모델

- `BridgeRoom`: 브릿지 방, 연결된 약속, 상태, 방장, 현재 단계
- `BridgeSeat`: 방별 좌석과 사용자 연결
- `BridgeDeal`: 딜러, 취약 상태, 셔플 seed 또는 서버 저장 딜 정보
- `BridgeCall`: 비딩 기록, 콜 순서, 패스/입찰/더블/리더블
- `BridgeTrick`: 트릭 단위 상태
- `BridgePlay`: 플레이된 카드 기록
- `BridgeEvent`: 방 상태 동기화용 이벤트 로그

## UI 방향

- 브릿지 로비는 기존 약속 화면의 참여 흐름을 최대한 재사용합니다.
- 브릿지 테이블 화면은 모바일보다 데스크톱/태블릿 사용성을 우선하되, 모바일에서 카드와 버튼이 겹치지 않게 합니다.
- 카드 UI는 이미지 52장이 아니라 `rank`, `suit` 데이터 기반 컴포넌트로 생성합니다.
- 턴, 현재 컨트랙트, 더미, 트릭 승자처럼 다음 행동 판단에 필요한 정보는 항상 화면에 보여줍니다.
- 플레이어 본인의 좌석은 브릿지 테이블 하단에 보이도록 좌석 표시를 회전합니다.
- 선언자 좌석은 별 표시로 구분하고, 취약 팀은 붉은 글자로 강조합니다.

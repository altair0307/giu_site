# 프로젝트 가이드라인

이 문서는 GIU 보드게임 대여 서비스의 기능 추가, 유지보수, 배포 전 확인 기준을 정리합니다. README가 프로젝트 소개와 실행 방법을 설명한다면, 이 문서는 실제 변경 작업에서 지켜야 할 판단 기준을 다룹니다.

## 기본 원칙

- 운영 데이터의 일관성을 우선합니다. 대여, 반납, 약속, 평점, 알림처럼 상태가 바뀌는 기능은 DB 상태 전이와 실패 시나리오를 먼저 확인합니다.
- 관리자 추적성을 유지합니다. 운영자가 나중에 누가, 언제, 무엇을 바꿨는지 확인해야 하는 행동은 로그를 남깁니다.
- 회원 화면은 빠른 검색과 요청을, 관리자 화면은 확인과 승인, 복구를 우선합니다.
- 새 기능은 기존 App Router, 서버 액션, Prisma, CSS 구성에 맞춰 추가합니다.
- 민감한 값은 코드와 문서에 넣지 않습니다. 환경변수 이름과 필요한 형식만 문서화합니다.

## 코드 구조 기준

- 회원용 주요 화면과 액션은 `src/app/page.tsx`, `src/app/actions.ts`를 중심으로 확인합니다.
- 관리자 화면은 `src/app/admin/*` 아래에 기능별 페이지로 둡니다.
- 약속 기능은 `src/app/meetups/*`와 `Meetup`, `MeetupParticipant`, `MeetupActivityLog` 모델을 함께 봅니다.
- 공통 서버 로직은 `src/lib/*`에 둡니다.
- DB 모델과 상태값은 `prisma/schema.prisma`에서 관리합니다.
- 보드게임 엑셀 처리는 `src/lib/game-spreadsheet.ts`에 모읍니다.
- Discord 알림과 알림 중복 방지는 `src/lib/notifications.ts`와 `NotificationLog`를 기준으로 합니다.

## 서버 액션과 검증

- 사용자 입력은 서버에서 Zod로 검증합니다.
- FormData 값은 trim 후 사용하고, 빈 문자열과 null 처리를 명확히 합니다.
- 로그인, 회원가입, 비밀번호 변경, 대여, 반납, 약속 생성처럼 반복 요청 위험이 있는 기능은 rate limit 적용 여부를 확인합니다.
- 실패 메시지는 사용자에게 이해 가능한 한국어로 반환합니다.
- 인증이 필요한 기능은 `requireUser()`를 사용합니다.
- 관리자 전용 기능은 현재 사용자 role 확인을 반드시 포함합니다.

## DB와 상태 전이

- 대여 가능 상태는 `Game.status`, 실제 대여 기록은 `Loan.status`, 승인 요청은 `LoanRequest.status`로 나뉩니다.
- 상태 전이를 추가할 때는 관련 모델이 함께 바뀌어야 하는지 확인합니다.
- 같은 게임을 동시에 대여하거나, 예정 약속이 있는 게임을 대여하는 상황을 막아야 합니다.
- 대여 중인 게임이 있을 때 전체 게임 DB 교체처럼 운영 데이터가 꼬일 수 있는 작업은 차단합니다.
- 모델 변경 시 Prisma migration을 만들고, 배포 DB에는 `npm run db:deploy`를 사용합니다.

## 운영 로그 기준

운영자가 추적해야 하는 성공 동작은 로그로 남깁니다.

- 대여/반납 이력은 `createLoanActivityLog`를 사용합니다.
- 약속 생성, 완료, 취소 이력은 `createMeetupActivityLog`를 사용합니다.
- 회원, 게임, 공지, 엑셀 import, 관리자 조작 등 일반 운영 이력은 `createGeneralActivityLog`를 사용합니다.
- 로그에는 actor, target, message, metadata를 가능한 한 구체적으로 남깁니다.
- 실패한 작업은 별도 장애 추적이 필요한 경우가 아니면 운영 로그보다 에러 처리와 사용자 메시지를 우선합니다.

## 알림 기준

- Discord 알림은 `sendManagedDiscordNotification`을 통해 보냅니다.
- 알림에는 중복 발송을 막는 `dedupeKey`를 둡니다.
- 알림 실패는 `NotificationLog`에 `FAILED`로 남기고, 기능 전체가 불필요하게 깨지지 않도록 처리합니다.
- 사용자에게 표시되는 날짜와 Discord 알림 날짜는 한국 시간 기준으로 보여줍니다.
- Railway Cron은 UTC 기준임을 문서와 설정에서 함께 확인합니다.

## 엑셀 데이터 기준

- 업로드와 다운로드 컬럼은 `GAME_HEADERS`를 기준으로 맞춥니다.
- 관리자 업로드는 첫 번째 시트의 A-K 컬럼을 읽습니다.
- 제목이 없는 행은 무시합니다.
- 존재 여부는 `ㅇ`, `o`, `ok`, `true`, `y`, `yes`, `x`, `false`, `n`, `no` 같은 값을 허용합니다.
- 신규 컬럼을 추가하면 README의 엑셀 양식, import, export를 함께 수정합니다.

## UI 기준

- 회원 화면은 보드게임 검색, 필터, 대여 요청을 빠르게 수행하는 데 집중합니다.
- 관리자 화면은 대시보드, 승인 대기, 대여 관리, 회원 관리, 약속 관리, 로그 관리처럼 기능별로 분리합니다.
- 상태값은 텍스트만 두지 말고 사용자가 다음 행동 가능 여부를 알 수 있게 버튼과 안내를 함께 정리합니다.
- 사진, 대여 상태, 반납 예정일처럼 운영 판단에 필요한 정보는 관리자 화면에서 바로 확인 가능해야 합니다.
- 모바일에서도 검색, 필터, 승인 버튼, 사진 미리보기가 겹치지 않는지 확인합니다.

## 보안과 개인정보

- 비밀번호는 bcrypt hash로만 저장합니다.
- 세션 토큰은 DB에 저장하고, 쿠키는 httpOnly, sameSite, production secure 설정을 유지합니다.
- 사진 데이터, 학번, loginId 등 개인 식별 가능 정보는 필요한 화면과 로그에만 노출합니다.
- `.env`, `.env*.local`, 업로드 원본 엑셀, 개인 인수인계 문서는 Git에 올리지 않습니다.
- 외부 Webhook URL은 환경변수로만 관리합니다.

## 배포 기준

- Railway 배포 전 환경변수 `DATABASE_URL`, `SESSION_SECRET`, `DISCORD_WEBHOOK_URL`을 확인합니다.
- DB migration이 있으면 배포 과정에서 `npm run db:deploy` 적용 여부를 확인합니다.
- standalone 배포 흐름은 `npm run build`와 `scripts/prepare-standalone-static.mjs`를 기준으로 유지합니다.
- 반납 지연 알림 Cron 서비스는 `railway.notify-loans.json` 설정을 확인합니다.
- 배포 후 관리자 계정으로 로그인, 대여 요청 승인, 반납 요청 알림, 관리자 로그 화면을 최소 점검합니다.

## 변경 전 체크리스트

- `git status --short`로 작업트리를 확인합니다.
- 관련 코드와 모델을 `rg`로 검색합니다.
- 회원 화면, 관리자 화면, 로그, 알림, DB 상태 중 어떤 영역에 영향이 있는지 적습니다.
- 새 환경변수가 필요한지 확인합니다.
- 운영 데이터 손상 가능성이 있으면 차단 조건이나 트랜잭션 필요 여부를 먼저 판단합니다.

## 변경 후 체크리스트

- `npm run lint`
- DB 변경 시 migration 파일 확인
- 사용자 입력 변경 시 검증 메시지 확인
- 상태 전이 변경 시 성공과 실패 흐름 수동 테스트
- 관리자 로그가 필요한 기능이면 로그 생성 여부 확인
- 알림 변경 시 dedupeKey와 실패 로그 확인
- UI 변경 시 모바일과 데스크톱 레이아웃 확인
